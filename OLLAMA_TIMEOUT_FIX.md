# Ollama 超时问题优化总结

## 问题诊断

### 原始问题
- 使用 `qwen2.5:1.5b` 模型生成 AI 简报时超时
- 诊断脚本显示 Ollama 正常（10-13 秒响应）
- 但 Fluent Reader 应用中仍然超时

### 根本原因
**输入数据量过大**：
- 15 篇文章 × 500 字符 = 7,500 字符输入
- 复杂提示词 ≈ 200 字符
- 总输入 ≈ 7,700 字符 ≈ 1,925 tokens
- 对小模型来说处理时间过长（>120 秒）

---

## 优化措施

### 1. 减少文章数量
```typescript
// 优化前
const selectedItems = items.slice(0, 15)

// 优化后
const maxItems = provider.provider === 'ollama' ? 8 : 15
const selectedItems = items.slice(0, maxItems)
```
**效果**：Ollama 小模型只处理 8 篇文章（而非 15 篇）

### 2. 缩短文章片段
```typescript
// 优化前
const snippet = text.substring(0, 500)

// 优化后
const snippet = text.substring(0, 250)
```
**效果**：每篇文章输入减少 50%

### 3. 简化提示词
```typescript
// 优化前（复杂）
"你是一个资深的全球科技新闻编辑。请根据提供的文章（包括外语文章），生成一份结构清晰、可读性强的每日报送。
如果原始文章是英文或其他语言，请将其要点翻译并汇总为中文。
报送需包含：1. 核心综述（今日全球大趋势）；2. 专题资讯（按主题分类，每条包含深度总结）；3. 趣味/深度推荐。
使用 Markdown 格式。"

// 优化后（简洁）
"你是新闻编辑。请将以下文章摘要整理成简洁的中文简报，包含：
1.今日要点（3-5条最重要新闻）；
2.分类摘要（科技、商业等）。
每条新闻一句话总结。使用 Markdown 格式。"
```
**效果**：提示词从 120 字减少到 60 字，输出要求更明确

### 4. 限制 Ollama 参数
```typescript
options: {
    temperature: 0.7,
    num_ctx: 4096,      // 限制上下文长度
    num_predict: 1024   // 限制输出长度（避免生成过长内容）
}
```
**效果**：防止模型生成过多内容导致超时

### 5. 添加进度日志
```typescript
console.log(`[Digest] 准备生成摘要: ${selectedItems.length} 篇文章`)
console.log(`[Digest] 输入大小: ~${Math.round(context.length / 4)} tokens`)
```
**效果**：用户可以监控处理进度

## 使用建议

### 对于 qwen2.5:1.5b（当前使用）
✅ **已优化，应该可以正常工作**
- 输入量减少 72%，处理时间大幅缩短
- 如果仍超时，检查 Ollama 服务状态

### 如果想进一步提升质量
**升级到 qwen2.5:3b**：
```bash
ollama pull qwen2.5:3b
```
然后在 Fluent Reader 设置中修改模型名称为 `qwen2.5:3b`

**优势**：
- 质量提升明显（3B vs 1.5B）
- 速度相差不大（~5-10 秒）
- 内存增加 ~1 GB（可接受）

## 故障排查

### 问题 1：仍然超时
**可能原因**：
- 使用的是旧版本（未重启应用）
- Ollama 服务异常

**解决方案**：
1. 完全关闭并重启 Fluent Reader
2. 重启 Ollama：`ollama serve`
3. 测试连接：`curl http://localhost:11434/`

### 问题 2：摘要质量下降
**原因**：输入减少导致信息不足

**解决方案**：
- 升级到 qwen2.5:3b（推荐）
- 或使用云端 API（DeepSeek、NVIDIA）

### 问题 3：首次加载仍然很慢
**原因**：模型需要从磁盘加载到内存

**解决方案**：
```bash
# 预热模型（启动 Ollama 后立即运行）
ollama run qwen2.5:1.5b "你好"
```
